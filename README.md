# BoothEnhanser

BOOTHで活動する購入者(主にVRChat向けアバター・衣装・パーツを探すユーザー)向けのブラウザ拡張機能。好きリストの管理拡張、アバター別検索、自動タグ付けなどを提供する。

売り手側(問い合わせ管理サービス)とは別プロジェクトとして独立して管理する。

> **注意/免責事項**
> 本拡張機能は個人による非公式のツールであり、pixiv株式会社・BOOTHとは一切関係ありません。BOOTHには公式の公開APIが存在しないため、本拡張機能はログイン中のセッション情報を用いてページ上の公開情報を解析する形で動作します。パスワード・セッション情報等を外部サーバーへ送信することはありません(自動タグ付け機能でLLM APIを利用する場合を除き、利用時はユーザー自身のAPIキーを使用します)。BOOTH側の仕様変更により予告なく動作しなくなる可能性があります。利用は自己責任でお願いします。

## 前提・制約

- BOOTHには公式の公開APIが存在しない。すべての機能はログインセッションを使った非公式アクセス(商品ページのJSON-LD等の解析)に依存する
- pixiv/BOOTHの利用規約との兼ね合いを踏まえ、自動送信・自動操作は行わず、閲覧・収集に留める
- UI変更でいつでも壊れうる前提で、疎結合な設計にする
- 実装形態: **ブラウザ拡張機能(Manifest V3)**。Firefox対応も視野に入れる場合はwebextension-polyfillで差異を吸収

## アーキテクチャ

```
- Content Script … BOOTHページへのUI注入・データ抽出
- Background (Service Worker) … 非同期処理、ストレージ管理、定期巡回
- Popup / Options Page … 設定画面、辞書編集、手動オーバーライド管理
- IndexedDB … 商品データ・ユーザープロファイルのローカルキャッシュ
```

## 機能一覧

### 1. 好きリストの管理拡張

**分類(3タブ構成)**
```
[アバター]           ← R-18分岐なし(アバター本体にR-18指定は実質存在しないため)
[アバター以外]
   └ [全年齢] [R-18]
[その他]              ← itemType判定不能なものの受け皿。手動修正の導線を用意
```

アイテム種別の判定はBOOTHのカテゴリ情報を一次情報とし、判定できない場合はタグ・説明文のキーワード辞書でフォールバックする。それでも判定できないものは`unknown`として「その他」タブに入れ、無理に二択へ押し込めない。

**並び替えオプション**
- 価格(安い順/高い順)
- 登録日(新しい順/古い順)
- 更新日(新しい順/古い順)
- ショップ名あいうえお順(`Intl.Collator('ja')`による近似、必要なら手動読み仮名オーバーライドで補正)
- 商品名あいうえお順(同上)
- ランダム(Fisher-Yatesシャッフル)
- 好み順(後述の好みプロファイルによる`affinityScore`)

**今日のピックアップウィジェット**
- 好きリスト上部に2〜3件、日付をシードにした疑似乱数(mulberry32等)で表示。同日中はリロードしても変わらない
- 登録から時間が経っている(＝忘れられている可能性が高い)アイテムほど出現しやすいよう対数的に重み付け
- 直近で表示済みのアイテムは`lastPickedAt`で管理し、一定期間は再出現を抑制してローテーションさせる
- 「スキップ」ボタンで即座に別候補へ差し替え可能

### 2. アバター別検索

BOOTHのタグ運用は出品者任せで表記揺れが激しい(「マヌカ」「Manuka」等)ため、単純なタグフィルタではなく、主要アバター名のエイリアス辞書を持ち、タグ・説明文の両方から対応アバターを正規化して抽出する。辞書はGitHubリポジトリで管理し、ユーザーがPRを送れる形にしてコミュニティ資産として育てる。

### 3. 自動タグ付け

説明文・タグから「衣装/髪型/アクセサリ/ギミック」「対応アバター」「PhysBone対応」「Modular Avatar対応」等を構造化する。まずルールベース(正規表現＋辞書)で大半をカバーし、精度が必要な部分だけLLM APIに投げるハイブリッド構成とする。APIキーはユーザー自身が設定する形とし、拡張側で秘密鍵を保持しない。

### 4. 購入・スキ傾向分析(好みプロファイル)

Amazon的な協調フィルタリングではなく、**ユーザー自身の購入・スキ履歴からタグ重みベクトルを作り、リスト内商品との類似度で採点するコンテンツベースフィルタリング**。

- 購入履歴(重み1.0)とスキ登録履歴(重み0.35)の両方からタグ重みを学習。購入は確定した好みとして強く、スキは弱いシグナルとして補強的に扱う
- 差分処理(前回処理済みIDからの続きのみ処理)により、都度全件再計算しない
- `profileVersion`をキーに商品側の`affinityScore`をキャッシュし、プロファイルが更新された時だけ再計算する
- プロファイル学習に使う全スキ履歴と、採点対象となる「現在の好きリスト内商品」は明確に分離する

## データモデル

```js
// items ストア
{
  itemId: string,
  itemName: string,
  shopName: string,
  shopId: string,
  price: number,
  registeredAt: string,
  updatedAt: string,
  tags: string[],
  avatarTags: string[],              // 正規化済み対応アバタータグ
  isAdult: boolean,                  // BOOTHのR-18フラグ
  itemType: 'avatar' | 'non-avatar' | 'unknown',
  itemTypeOverride: 'avatar' | 'non-avatar' | null,  // 手動修正
  affinityScore: number | null,
  scoredWithProfileVersion: number | null,
  yomiOverride: { shopName?: string, itemName?: string },  // 手動読み仮名
  lastPickedAt: string | null        // 今日のピックアップでの直近表示日時
}

// userProfile ストア(1レコードのみ)
{
  id: 'main',
  tagWeights: { [tag: string]: number },
  lastPurchaseIdProcessed: string,
  lastFavoriteIdProcessed: string,
  profileVersion: number,
  updatedAt: string
}
```

## MVPスコープ(フェーズ分け)

**フェーズ1(MVP)**
- データ収集基盤(商品ページ巡回・IndexedDBキャッシュ)
- 3タブ分類(アバター/アバター以外/その他)+ R-18サブタブ
- 並び替え(価格・日付・あいうえお・ランダム)
- 今日のピックアップウィジェット

**フェーズ2**
- アバター別検索(辞書ベース)
- 自動タグ付け(ルールベースのみ、LLM連携は後回し)

**フェーズ3**
- 好みプロファイル・affinityScoreによる「好み順」ソート(購入/スキ履歴が一定量溜まってから効果を発揮)
- LLMハイブリッドのタグ付け精度向上

**フェーズ4(仕上げ)**
- 手動オーバーライドUI(読み仮名・アイテム種別)

フェーズ1が全機能の土台となるため最優先。フェーズ3は履歴データの蓄積に依存するため、公開後の運用の中で本領発揮する位置づけ。

## 売り手側プロジェクトとの関係

売り手側(問い合わせ管理サービス、GitHub Issue/Google Form連携、サブスク型)は完全に別プロジェクト・別リポジトリとして管理する。技術的な依存関係はないが、将来的に出品者ダッシュボードから買い手側の辞書データへ情報提供する等の連携可能性はある。買い手側=無料OSSで信頼構築、売り手側=有料SaaSで収益化、という役割分担とする。

## ソフトウェアライフサイクル・運営方針

**ブランチ戦略**: `main`(安定)+ `develop`(開発中)のシンプルな構成。

**CI/CD**: GitHub Actionsで`manifest.json`のバリデーション・Lint・ビルド確認を実施。リリースはSemVer(`major.minor.patch`)に従い、Chrome Web Store / AMOへは手動または自動提出ツール(`chrome-webstore-upload-cli`等)を使う。

**壊れやすさへの備え**: BOOTHのDOM構造・JSON-LDのキー名変更で拡張が壊ることを前提とし、抽出ロジック(セレクタ等)は一箇所に集約して緊急パッチを出しやすくする。バグ報告は拡張内にフォームを設置し、GitHub Issueへ起票する導線を用意する(売り手側で設計した起票の仕組みを流用)。

**辞書のコミュニティ運営**: アバターエイリアス辞書・キーワード辞書はPRベースでの更新を想定。マージ権限者を明確にしておく。

**EOL**: 個人開発のため、メンテナンス休止の可能性を正直にREADMEへ明記する。休止時はChrome Web Store側のステータスにも反映する。
